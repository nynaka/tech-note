# Gemini CLI を使った技術情報管理

本ドキュメントでは、Gemini CLI を使用して Docusaurus 上の技術ナレッジを効率的に作成・レビュー・修正するための初期設定および運用手順をまとめます。

---

> **レビューノート（2026年4月26日）**: 元のドキュメントから修正・補足した主な点は以下の通りです。
>
> 1. **`/memory reload` → `/memory refresh`**: 正式なコマンドは `refresh` です（`reload` は非対応）
> 2. **JIT 読み込みの説明補足**: サブディレクトリの `GEMINI.md` は「アクセス時」ではなく「起動時に全スキャン」される（JIT はデフォルト有効だが設定で変更可）
> 3. **`@` インポート構文の補足**: `@path/to/file.md` 形式が正式（`@include` ではない）
> 4. **スキルの同名優先順位**: Workspace > User > Extension の順。さらに同一ティア内では `.agents/skills/` が `.gemini/skills/` より優先
> 5. **スキル有効化のフロー補足**: ユーザーへの確認プロンプトが表示される点を明記
> 6. **`/skills disable|enable` のスコープ**: デフォルトは `--scope user`。ワークスペース対象には `--scope workspace` が必要
> 7. **ポリシーのワークスペース単位の状況更新**: v0.31.0（2026-02-27）でワークスペースポリシーが実装されたが、執筆時点でバグ報告あり（信頼済みフォルダでも未ロードのケースが多い）。ユーザー単位が安定
> 8. **`/plan` の正確な説明**: `/plan` はスラッシュコマンドで Plan Mode のトグル。「プランの調査・抽出」はモード内での自然言語指示で行う
> 9. **`/init` コマンドの追記**: プロジェクト用の `GEMINI.md` 雛形を自動生成する便利なコマンド
> 10. **`commandPrefix` フィールドの補足**: ポリシーの条件フィールドは `commandPrefix` ではなく `argsPattern`（正規表現）が正式。`commandPrefix` も利用可能なケースはあるが要確認

## 1. 効率的な作業のための初期化

Gemini CLI は、プロジェクト直下の設定ファイルを参照して挙動を最適化します。

### 恒久的な指示の定義 (`GEMINI.md`)
`GEMINI.md` を配置することで、エージェントに対して常に守らせたい基本原則（表記揺れの防止、ディレクトリ構造の維持、技術スタックの前提など）を指示します。

- **役割**: 全てのセッションで自動的に読み込まれる「基本命令」。
- **記述例**:
  ```markdown
  - ドキュメントはすべて Markdown 形式で作成すること。
  - Docusaurus のディレクトリ構造 (`docs/`, `static/`, `sidebars.ts`) を尊重すること。
  - 日本語の文章は丁寧語（です・ます）を基本とする。
  ```

> **ヒント**: 新しいプロジェクトで `GEMINI.md` を一から書くのが大変な場合は、セッション内で `/init` を実行すると、現在のディレクトリを解析してプロジェクト固有の雛形を自動生成してくれます。

#### 読み込み階層
複数の場所に `GEMINI.md` を配置でき、すべてが結合されてエージェントに渡されます。読み込み順序は以下の通りで、後に読み込まれるファイル（より具体的なもの）が上書き・補完します。

| 優先順位 | パス | 用途 |
|---------|------|------|
| 低 | `~/.gemini/GEMINI.md` | 全プロジェクト共通のルール |
| 中 | `<プロジェクトルート>/GEMINI.md`（`.git` フォルダを境界に上位ディレクトリも探索） | リポジトリ固有のルール |
| 高 (JIT) | `<サブディレクトリ>/GEMINI.md` | 特定ディレクトリ専用のルール（デフォルトでは起動時に全サブディレクトリをスキャンしてロード。JIT モードは設定で無効化可能） |

> **注意**: JIT（Just-In-Time）コンテキスト読み込みはデフォルトで有効です。設定ファイルの `context.jitContextLoading` を `false` にするとサブディレクトリの `GEMINI.md` を起動時にまとめてシステムプロンプトへ展開する動作に切り替わります。フッターに「読み込まれたコンテキストファイル数」が表示されるため、現在の状態を視覚的に確認できます。

#### ファイル分割 (`@` インポート)
`GEMINI.md` が肥大化した場合、`@path/to/file.md` 構文で別ファイルに分割できます。

```markdown
# Main GEMINI.md
@./docs/style-guide.md
@./docs/review-checklist.md
```

#### `/memory` コマンドでコンテキストを管理する

セッション中に以下のスラッシュコマンドでコンテキストを操作できます。

| コマンド | 説明 |
|---------|------|
| `/memory show` | 現在読み込まれている全指示内容を表示（デバッグに有効） |
| `/memory refresh` | `GEMINI.md` を手動で編集した後に再読み込み ⚠️ **`/memory reload` は誤り** |
| `/memory add <テキスト>` | `~/.gemini/GEMINI.md` に永続メモを追記 |

> **補足**: セッション中に `GEMINI.md` を編集しても、エージェントはすぐに変更を認識しません。`/memory refresh` で強制的に再スキャン・再読み込みを行ってください。

### ポリシー設定 (`~/.gemini/policies/`)
シェルコマンドの実行許可などをユーザー単位で制御したい場合は、`~/.gemini/policies/` 配下に任意の `.toml` ファイルを作成します。同ディレクトリの `.toml` ファイルはすべて結合されて読み込まれます。

```bash
mkdir -p ~/.gemini/policies
```

```toml
# ~/.gemini/policies/my-rules.toml
[[rule]]
toolName = "run_shell_command"
argsPattern = "rm -rf.*"
decision = "deny"
priority = 100
```

> **ポリシーの優先順位 (Tier)**
>
> | Tier | 場所 | 説明 |
> |------|------|------|
> | 1（低） | 組み込み | Gemini CLI デフォルトポリシー |
> | 2 | `.gemini/policies/`（ワークスペース） | プロジェクト固有（※後述の注意参照） |
> | 3 | `~/.gemini/policies/`（ユーザー） | ユーザー全プロジェクト共通 |
> | 4（高） | システム管理者ディレクトリ | 企業・管理者向け強制ポリシー |
>
> 数値が高いほど優先度が高く、低い Tier のルールを上書きします。

> **⚠️ ワークスペース単位ポリシーの現状 (2026年4月時点)**
>
> ワークスペース単位のポリシー（`.gemini/policies/`）は v0.31.0 で実装されましたが、**信頼済みフォルダに配置しても読み込まれないバグが複数報告されています**（Issue #18186, #21580, #21598）。現時点では **ユーザー単位（`~/.gemini/policies/`）のみ安定動作します**。ワークスペースポリシーを使う場合は最新バージョンで動作確認を行ってください。

## 2. スキル (Skill) の作成と配置

スキルは、特定の専門タスク（例：技術レビュー、特定のフレームワークのコード生成）をエージェントに実行させるための「専門知識パッケージ」です。初回はコンテキストウィンドウに影響しない軽量なメタデータ（`name` と `description`）のみが読み込まれ、タスクに一致した際にフルの指示が展開されます（Progressive Disclosure）。

### 配置場所
- **ワークスペース固有**: `.gemini/skills/<skill-name>/SKILL.md`（または `.agents/skills/` エイリアス）
  - バージョン管理に含めてチームで共有する用途に適しています。
- **ユーザー共通**: `~/.gemini/skills/<skill-name>/SKILL.md`（または `~/.agents/skills/` エイリアス）
  - 全プロジェクトで利用できます。

> **優先順位**: 同名スキルが複数の場所に存在する場合、**Workspace > User > Extension** の順で優先されます。同一ティア内では `.agents/skills/` が `.gemini/skills/` より優先されます。

### 作成方法

#### 方法 A: `skill-creator` スキルを使う（推奨）
Gemini CLI の対話セッション内で以下のように依頼するだけで、スキルの雛形が自動生成されます（`SKILL.md` に加えて `scripts/`、`references/`、`assets/` 等の標準ディレクトリ構造も生成）。

```text
> 「tech-reviewer」という新しいスキルを作成して
```

#### 方法 B: 手動作成
1. **ディレクトリ作成**: `.gemini/skills/tech-reviewer/` のようなディレクトリを作成します。
2. **`SKILL.md` の作成**: 以下の形式で記述します（YAML フロントマターの `name` と `description` は必須）。

```markdown
---
name: tech-reviewer
description: Markdownドキュメントの技術的な正確性と構成をレビューするスキル
---

# 技術レビュー・プロトコル
エージェントはこのスキルが有効化された際、以下のチェックリストに従ってレビューを実施します。
1. Docusaurus の Frontmatter (title, id 等) が正しく設定されているか。
2. コードブロックに言語指定がされているか。
3. 手順書の場合、前提条件が明記されているか。
```

> **スキル有効化の流れ**
> 1. Gemini がタスクに一致するスキルを検出すると `activate_skill` ツールを呼び出します。
> 2. UI に確認プロンプトが表示され、スキル名・用途・アクセスを許可するディレクトリパスが提示されます。
> 3. 承認すると `SKILL.md` の本文とフォルダ構造がコンテキストに読み込まれ、スキルのディレクトリが許可済みファイルパスに追加されます。
> 4. スキルはセッション終了まで有効です。

### スキルの管理
対話セッション内でスラッシュコマンドを使って管理します。

```text
/skills list               # 利用可能なスキルを一覧表示
/skills reload             # スキルの再スキャン
/skills disable <名前>     # 特定スキルを無効化（デフォルトはユーザースコープ）
/skills disable <名前> --scope workspace  # ワークスペーススコープで無効化
/skills enable <名前>      # 特定スキルを有効化
```

> **`/skills disable|enable` のスコープ**: デフォルトは `--scope user`（ユーザーレベル）です。ワークスペース固有のスキルを操作する場合は `--scope workspace` を明示してください。

ターミナルからも管理できます。

```bash
gemini skills list
gemini skills install https://github.com/user/repo.git  # Git リポジトリからインストール
gemini skills install /path/to/local/skill              # ローカルからインストール
gemini skills uninstall my-skill --scope workspace
```

### 利用方法
スキルの `description` に合致するタスクをエージェントが検出した際に自動で呼び出されます。または、プロンプトで明示的に指示することもできます。

```text
> tech-reviewer スキルを使ってこのファイルをレビューして
```

## 3. 既存 Markdown からのチェックリスト作成手順

既存の優れたナレッジファイルから「レビュー時のチェックリスト」を抽出してスキル化する効率的な手順です。

### 手順 A: Plan Mode による安全な調査・抽出

`/plan` はセッション内で **Plan Mode（読み取り専用モード）をトグルするスラッシュコマンド**です（または `Shift+Tab` でも切り替え可）。Plan Mode 中はファイルの読み取り・調査は自由に行えますが、書き込みはプランファイルへの出力のみに制限されます。

1. **セッション開始**: `gemini` を起動してプロジェクトルートに入ります。
2. **Plan Mode に切り替え**:
   ```text
   /plan
   ```
   またはセッション内でエージェントに依頼することでもモードが切り替わります：
   ```text
   > docs/linux/ のドキュメントから共通する構成ルールを抽出するプランを作って
   ```
3. **調査と抽出**: Plan Mode 内でエージェントが安全に既存ファイルを読み込んでプラン（チェックリスト案）を提示します。デフォルトでは `~/.gemini/tmp/<project>/<session-id>/plans/` にプランファイルが出力されます。
4. **Plan Mode を終了してスキル化**: プランを承認・確認後、通常モードに戻ってから抽出されたルールを `.gemini/skills/document-checker/SKILL.md` に保存するよう指示します。

> **プランファイルの保存先をプロジェクト内にしたい場合**は `settings.json` でカスタム保存ディレクトリを設定し、対応するポリシーファイル（書き込み許可）を追加してください。

### 手順 B: `GEMINI.md` へのフィードバック
特定のファイルだけでなく、プロジェクト全体で守るべきルールが見つかった場合は、それを `GEMINI.md` に追記することで、スキルの呼び出しなしに常時適用されるようになります。

## 4. 推奨ワークフロー

### 対話セッションでの作業（推奨）

```bash
cd /path/to/tech-note
gemini
```

セッション内で自然言語で指示します。

```text
> draft/kcna-jp.md の内容をベースに、新しい手順書を docs/linux/ に作成して
> 作成したファイルを tech-reviewer スキルを使ってチェックし、修正案を提示して
> 全て適用して
```

### 非対話モード（スクリプト・ワンライナー）

`-p` フラグを使うことでスクリプトからも呼び出せます。

```bash
gemini -p "docs/linux/ 配下の全ファイルに Frontmatter の title が設定されているか確認して、不足しているファイルを一覧表示して"
```

> **非対話モードの Plan Mode 動作**: 非対話環境（CI/CD 等）では Plan Mode の enter/exit が自動承認され、プラン承認後は YOLO モード（全ツール自動承認）で実装が実行されます。CI での利用時はこの挙動を考慮してください。

| `/skills enable/disable <名前>` | スキルの有効・無効を切り替え |
| `/policies list` | 有効なポリシールール一覧を表示（デバッグに有効） |
| `/settings` | 各種設定を変更 |
