属性ベース暗号 (Attribute-Based Encryption: ABE)
===

## 1. 概要：従来の暗号との決定的な違い

RSA や ECC が「特定の公開鍵 (=特定の個人/エンティティ)」を宛先にして暗号化するのに対し、ABEは「属性の集合や論理条件 (ポリシー)」を宛先にして暗号化します。

| 項目         | 従来の公開鍵暗号 (RSA/ECC)             | 属性ベース暗号 (ABE)                                         |
| :----------- | :------------------------------------- | :----------------------------------------------------------- |
| 対象         | 1 対 1 (Identity-based)                | 1 対多 / 条件ベース (Policy-based)                           |
| アクセス制御 | 復号後にアプリ側で制御                 | 暗号文自体に数学的な制御が内包される                         |
| 鍵の管理     | 個人の公開鍵を事前に取得する必要がある | 送信者は「属性の条件」を知っていればよく、個々の公開鍵は不要 |
| 数学的基盤   | 素因数分解、離散対数問題               | ペアリング演算 (双線形写像)                                  |

### 技術的補足：ハイブリッド暗号方式

実運用では、AES等の対称鍵暗号と組み合わせて使用されます。  
大きなデータ本体はAES (データカプセル化メカニズム：DEM) で暗号化し、その AES共通鍵をABEで暗号化 (鍵カプセル化メカニズム：KEM) します。

---

## 2. 方式の分類と詳細

ABEには、ポリシーを「暗号文」に持たせるか「秘密鍵」に持たせるかで2つの主要な方式があります。

### CP-ABE (Ciphertext-Policy ABE)

「暗号文にポリシー」を埋め込む方式。  
現実の組織におけるアクセス制限 (例：「マネージャー」かつ「人事部」のみ閲覧可) に直結するため、最も一般的に利用されます。

- **鍵:** ユーザーは自身の属性 (Role: Manager、Dept: HR) に基づいた「属性秘密鍵」を持つ。
- **暗号文:** データの送信者が、論理式 (OR, AND、Threshold) によるアクセス構造を付与して暗号化する。

### KP-ABE (Key-Policy ABE)

**「秘密鍵にポリシー」を埋め込む方式:** データ自体には「属性タグ」を付与し、誰がどの条件のデータにアクセスできるかを「鍵発行時」に決定します。

- **暗号文:** 複数の属性タグ (例：「重要度：高」「カテゴリ：ログ」「年度：2024」) を付与して暗号化。
- **鍵:** ユーザーの鍵にポリシー (例：「重要度：高」AND「カテゴリ：ログ」) が埋め込まれている。

### MA-ABE (Multi-Authority ABE) ※追加項目

CP-ABE / KP-ABEでは鍵発行機関 (TA) が単一であり、TAへの信頼集中 (Single Point of Trust) が課題となります。  
MA-ABEはこの問題に対処するため、複数の独立した認証機関 (Authority) が協調して鍵を発行する方式です。

- 各認証機関は自分のドメインの属性（例：「人事部」はHR機関、「開発部」は技術機関が管理）を管理する。
- ユーザーは複数機関から部分鍵を取得し、それらを組み合わせて復号に利用する。
- ゼロトラスト環境やフェデレーション型システムとの親和性が高い。

## 3. 暗号・復号シーケンス

### CP-ABE (Ciphertext-Policy ABE)

1.  セットアップ (Setup)
    - 信頼できる第三者機関 (TA) が、システム全体の **公開パラメータ (Public Parameter: PP)** と、管理用の **マスター鍵 (Master Key: MK)** を生成します。

2.  鍵発行 (KeyGen)
    - TAがユーザーの属性集合 (例：**\{開発部, チームA\}**) を確認し、MKを用いてそのユーザー専用の **属性鍵 (SK)** を生成・配布します。

3.  暗号化 (Encrypt)
    - 送信者が PP と、アクセス条件（例：**"開発部" AND ("マネージャー" OR "チームA")**）を用いてメッセージを暗号化します。

4.  復号 (Decrypt)
    - 受信者が持つ属性鍵の属性セットが、暗号文のアクセス構造（ポリシー）を満たす場合のみ、ペアリング演算によって数学的に復号できます。

> ⚠️ **修正点:** 元文の「暗号文を解凍できます」という表現は不正確です。ABEの復号は「展開 (解凍)」ではなく、ペアリング演算によりランダム化された値が打ち消し合い、平文 (またはAES鍵) が **数学的に復元** されるプロセスです。

### KP-ABE (Key-Policy ABE)

1. セットアップ (Setup)
    - TAが **公開パラメータ (PP)** と **マスター鍵 (MK)** を生成します。

2. 暗号化 (Encrypt)
    - 送信者がデータに対し、属性のリスト（例：**\{文書種別: 契約書, 秘匿度: 極秘, 拠点: 東京\}**）を付与し、PP を用いて暗号化します。

3. 鍵発行 (KeyGen)
    - TAがユーザーに対し、特定のアクセス条件（例：**"文書種別: 契約書" AND "拠点: 東京"**）を埋め込んだ **ポリシー付き秘密鍵 (SK)** を発行します。

4. 復号 (Decrypt)
    - 暗号文に付与された属性リストが、ユーザーの鍵に埋め込まれたポリシーを充足する場合のみ、復号が可能になります。

## 4. 技術的な注意点：ペアリング演算

ABE の実装を理解する上で避けて通れないのが **ペアリング (双線形写像)** です。  
RSA が Z_n^* 上のべき乗、ECC が楕円曲線上の点 P のスカラー倍 kP を使うのに対し、ABE は「2 つの群の要素を掛け合わせると別の群の要素に移る」という特殊な性質を利用します。

ペアリング **e: G₁ × G₂ → G_T** は以下の双線形性を満たします：

```
e(aP, bQ) = e(P, Q)^(ab)
```

この性質により、秘密鍵側の要素と暗号文側の要素を複雑に組み合わせ、アクセス条件が一致したときのみ値が打ち消し合って平文が取り出せるよう設計されています。

> ⚠️ **修正点:** 元文の e(P, Q) = e(Q, P) という式は「対称ペアリング（Type-1）」の場合のみ成立する性質（G₁ = G₂ のとき）です。一般的なペアリングの本質的な性質は双線形性（上式）であり、対称性は特殊ケースです。ABEの解説としては双線形性を主軸に説明すべきです。

### 計算コストについて

ペアリング演算は RSA や ECC の基本演算と比べて計算コストが高く、復号時の処理時間がボトルネックになりやすい点に注意が必要です。  
属性数やポリシーの複雑度に比例して演算回数が増加します。

## 5. セキュリティ上の重要概念

### 共謀耐性 (Collusion Resistance)

ABEの最重要なセキュリティ要件です。異なるユーザーが自分の属性鍵を持ち寄っても、それぞれ単独では復号できないポリシーを満たすことはできない という性質です。

例：ポリシーが "A属性" AND "B属性" のとき、A属性のみ持つユーザーとB属性のみ持つユーザーが鍵を合わせても復号できない設計になっています。これはKeyGen時に各ユーザーの鍵にランダム化（ブラインディング）を施すことで実現されます。

### 選択的安全性 vs 適応的安全性

| 安全性モデル                      | 概要                                                                              |
| :-------------------------------- | :-------------------------------------------------------------------------------- |
| 選択的安全性 (Selective Security) | 攻撃者が攻撃対象のポリシーを事前に宣言する（弱いモデル）。初期のABE方式はこちら。 |
| 適応的安全性 (Adaptive Security)  | 攻撃者がいつでも対象を変更できる（強いモデル）。実用上はこちらが望ましい。        |

## 7. 既知の課題と研究動向

### 現在の主な課題

- **計算コスト:** ペアリング演算はECCより数倍〜数十倍遅く、IoT や低スペックデバイスへの適用が難しい。
- **鍵失効 (Key Revocation):** 属性が変化したユーザーの鍵を無効化する仕組みが複雑で、標準化されていない。
- **鍵の漏洩責任追跡:** 誰が鍵を漏洩したかを追跡する「説明責任 (Accountability)」の仕組みが必要な場合がある。

### 研究動向

- **格子ベースABE:** ペアリングを使わず格子暗号（LWEなど）に基づくABEの研究が進んでおり、耐量子計算機暗号としての応用が期待されている。
- **関数型暗号 (Functional Encryption) との関係:** ABEは関数型暗号の特殊ケースとして位置づけられており、より広範な計算を秘密鍵に埋め込める関数型暗号の研究と密接に関連している。

## まとめ：RSA/ECCとの使い分け

- **RSA/ECCを選ぶべき時:** 宛先が1つに決まっており、高速な処理が必要な場合。
- **ABEを選ぶべき時:** 「誰が読むか」を暗号化時点で特定せず、属性に基づいた柔軟な権限管理 (ゼロトラスト環境やクラウドストレージの共有設定など) を暗号レイヤーで実現したい場合。
- **MA-ABEを選ぶべき時:** 組織横断的なシステムや、単一のTAへの信頼集中を避けたい分散型アーキテクチャを採用する場合。
